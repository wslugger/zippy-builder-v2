# AI & Prompting Standards

These rules dictate how we handle LLM and AI logic, based directly on past production wins and issues.

- **Strict Schema Enforcement**: Always give the exact JSON structures and active metadata values in the prompt (e.g., "Use only these Interface Types: [1GE, 10GE...]").
- **Vendor-Specific Context**: Split prompt instructions by vendor when necessary. (e.g., "Forwarding rate" vs "Throughput").
- **Server-Side API**: Keep AI interactions in Server Actions (`/api/*`) rather than Client components to protect keys and prevent browser interference.
- **Tolerant Data Binding**: UI components consuming AI-extracted data should intelligently fall back across expected variations of a property name (e.g. `item.specs.wifi_standard || item.specs.wifiStandard`). The AI pipeline is occasionally loose with snake_case vs camelCase.
- **Extracting Core Attributes**: Be explicit. When asking the AI to parse notes, explicitly enforce that it extracts mandatory core features (like `bandwidth`, `ports`, `redundancyModel`). Don't leave critical fields up to "soft" derivation.
- **Human-in-the-Loop Validation**: For rules and policies generated by "Copilot" AI, ensure a verification gate. Do not blind-save AI text to the database without presenting it to the user.
- **System Filtering Constraints**: When dynamically generating rules, ensure the AI's output is wrapped or anchored with mandatory system filters (e.g., applying `serviceId === 'managed_lan'` when making a LAN rule).
